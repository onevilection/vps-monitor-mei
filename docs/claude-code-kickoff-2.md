## A. `ts` を「履歴軸の正」として明記（最重要・方針変更）

これまで履歴は「到着順＝1Hz等間隔の配列インデックス」で時刻を表す前提だったが、ネット遅延・欠落・再接続でズレるため、**`ts`（サーバが各サンプルに打刻した時刻）を時間軸の正**に変更する。

- `docs/ndjson-schema.md` に以下を明記：
  - `ts` の**単位＝Unix epoch 秒・UTC**、サーバ打刻。
  - `ts` は**履歴軸の正**。client は各サンプルを **`ts` に基づくスロット**へ配置する（到着順では並べない）。
  - **欠落＝穴**：その `ts` のサンプルが来なければスロットは空（＝欠測 `null`）のまま。グラフはその区間で線を途切れさせる（または点線）。**左詰めにしない**。
  - **遅延・バースト＝再配置**：まとめて届いても各サンプルを自分の `ts` のスロットへ置く。
  - **重複＝統合**：同一 `ts` は1つにまとめる（後勝ち、または無視）。
- `docs/design.md` §5.6.2 を改訂：
  - 「到着順の純リングバッファ」という記述を「**`ts` 基準のスロット割り付け**（固定長3600スロット、`slot = ts − t0`）」に置換。
  - **メモリ設計は不変**（固定長・タイムスタンプ全持ちはしない）。変えるのは「割り付けのキーを到着順→`ts`」だけ、と明記。
  - 値が来ていないスロットは「**欠測（null）**」として、データのある区間と明確に区別する。
- 時計ずれの扱い：**サーバの `ts` をそのまま採用（サーバ時刻基準で表示）**。各パネルは独立なので複数サーバ間の時刻整合は不要、と一文添える。

## B. 接続断検出は「スキーマ外」と明記（責務の切り分け）

接続断（「サーバーに繋がらないよ」）は `ts` とは**別系統の検出**である。混同しないよう線引きを明記する。

- `docs/design.md` §5.4 に明記：
  - 接続断は **client 内部の wall-clock ウォッチドッグ**で検出する（最終受信からの経過時間。SSH の EOF／例外で即時 `Disconnected`、無音が閾値秒〔例10秒〕継続でも `Disconnected`）。
  - **`ts` では生死判定しない**（断中は新しい `ts` が来ない／時計ずれが混入するため）。判定は純粋にローカル時計で行う。
  - **NDJSON スキーマには載せない**：エージェントはステートレスで、落ちたら何も送れない（自己申告不可）。接続断は完全にクライアント状態。
  - 履歴との接点：断中はサンプルが来ないため該当スロットは穴（null）として残り、復帰後のグラフは欠測区間が途切れ／点線で表現される。検出ロジックは別だが見え方は一貫する。
- `docs/ndjson-schema.md` 側にも「接続断はスキーマ非依存（client 側 wall-clock 検出）」と1行注記し、スキーマに断のフィールドを足さないことを明示。

## C. fixture を追加（null と複数 disk）

代替値（ダミーの数字）は**入れない**。「測れないときは `null`（＝測定中／欠測）」を client が正しく扱えることを、契約段階のテスト見本で担保する。

- `testdata/sample_measuring.ndjson` を新規作成：
  - レート系 `cpu_pct` / `net.rx_bps` / `net.tx_bps` が **`null`** の「測定中」行（起動直後の1周目を表す正常な状態）。
  - 実IPなし・ダミーid。JSON 妥当性を検証。
- `testdata/sample_multidisk.ndjson` を新規作成：
  - `disk` が**2要素**（`"/"` と `"/var/www"`）の行。client が配列を正しく回せることの担保。
  - JSON 妥当性を検証。
- `docs/ndjson-schema.md` の該当節から3系統の fixture を参照し、**「正常」「測定中(null)」「複数disk」がテストの参照点**であることを明記。
- null セマンティクスを明記：`null` は「異常」ではなく「まだ測れていない／この回は測れない」正常状態。client は **数値として扱わず欠測（穴）として扱い、しきい値判定をスキップ**する。代替値での穴埋めは禁止。
